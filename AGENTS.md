# Project Agent Rules

## 목표

이 프로젝트에서 Codex는 작업의 불확실성에 따라 `가벼운 분석 -> 단계 계획 -> 실행 -> 기록` 순서로 진행한다.
모든 사용자-facing 응답과 프로젝트 산출물은 한국어로 작성한다.

## 기본 진행 원칙

1. 단순하고 범위가 명확한 요청은 질문 없이 바로 구현한다.
2. 요청이 모호하거나 범위가 큰 경우에만 `lightweight-discovery`를 먼저 수행한다.
3. 구현이 2개 이상의 파일 또는 모듈에 걸치거나 여러 단계가 필요하면 `phase-planner`를 사용한다.
4. 계획된 phase를 실행할 때는 `phase-executor`를 사용하고, 실행 후 `docs/ai-work/history/`에 변경 요약, 결정사항, 검증 결과를 짧게 기록한다.
5. 질문은 최소화한다. 작은 수정은 질문 없이 진행하고, 중간 수정은 핵심 질문 최대 1개, 큰 기능이나 구조 변경은 최대 3개까지만 질문한다.
6. 질문 없이 합리적으로 진행 가능한 경우에는 best-effort로 진행한 뒤, 가정과 검증 결과를 기록한다.

## 로컬 Skill 사용 기준

- `lightweight-discovery`: 모호하거나 큰 요청을 실행 가능한 요약으로 좁힐 때만 사용한다.
- `phase-planner`: 2개 이상의 파일/모듈을 건드리거나 여러 단계가 필요한 작업을 계획할 때 사용한다.
- `phase-executor`: `docs/ai-work/phases/phase-NNN-*.md`에 정의된 phase를 실행할 때 사용한다.
- 프로젝트 전용 추가 skill은 위 3개로 제한한다. oh-my-codex 기본 플러그인과 기본 skill은 필요할 때 그대로 사용할 수 있다.

## 산출물 위치

- 설계와 결정사항: `docs/ai-work/01-decisions.md`
- 단계 계획: `docs/ai-work/02-roadmap.md`
- phase 문서: `docs/ai-work/phases/phase-NNN-*.md`
- 세션 로그: `docs/ai-work/history/YYYY-MM-DD-session.md`

## TDD와 검증

버그 수정, 핵심 비즈니스 로직 변경, 리팩토링, 레거시 동작 보존, 파서/변환기/분석기/코드 생성기/CLI 구현처럼 기존 동작을 깨뜨릴 위험이 있는 작업은 가능한 한 TDD 흐름을 따른다.

1. 현재 동작 또는 기대 동작을 먼저 설명한다.
2. 실패하는 테스트를 먼저 작성한다.
3. 테스트가 실제로 실패하는지 확인한다.
4. 테스트를 통과시키는 최소 구현을 한다.
5. 새 테스트와 관련 기존 테스트를 실행한다.
6. 중복, 복잡도, 이름을 정리한다.
7. 변경 이유와 검증 결과를 history에 기록한다.

문서 수정, 설정값 변경, 단순 오타 수정, UI 문구 변경, 실험용 프로토타입, 테스트 환경이 없는 초기 스캐폴딩은 테스트 파일을 먼저 만들지 않아도 된다. 이 경우에도 수동 검증 방법이나 확인 기준을 history에 남긴다.

## 레거시 안전 규칙

레거시 코드는 기존 구현을 SSOT로 취급한다.

1. 기존 동작을 먼저 관찰한다.
2. 의도가 불명확하면 현재 동작을 보존하는 characterization test를 먼저 작성한다.
3. 이상해 보이는 동작도 변경 의도가 명확해지기 전까지는 현재 동작으로 고정한다.
4. 변경 전/후 동작 차이를 명시한다.
5. 의도적으로 동작을 바꾸는 경우에는 이유를 `docs/ai-work/01-decisions.md`에 기록한다.

## 완료 기준

작업을 완료했다고 보고하기 전에 다음을 확인한다.

- 요청 범위가 충족되었는가
- 필요한 테스트, 빌드, 정적 확인 또는 수동 검증이 수행되었는가
- 검증하지 못한 부분과 남은 위험을 명시했는가
- 필요한 결정사항과 세션 로그를 한국어로 기록했는가
